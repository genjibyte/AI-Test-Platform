# TestAgent Lab 项目规划文档

> 本文档是 `00_PROJECT_CHARTER.md` 的下级文档。
> 本文档只做**规划**：阶段划分、里程碑、任务拆解、验收映射、风险与排期。
> 本文档**不做**代码架构、技术选型、接口设计、数据库设计。这些留待《技术架构文档》。
> 若本文档与 Charter 冲突，以 Charter 为准。

---

## 1. 规划目标

把 Charter 第 9 节「MVP 成功标准」的 14 条，转化为**可执行、可验收、有顺序、有依赖**的工程计划。

规划遵循三条原则：

1. **闭环优先**：先把最窄的一条「能跑通的链路」打通，再加宽能力。
2. **可验收**：每个阶段都对应 Charter 中的明确指标，不靠主观判断完成与否。
3. **不超纲**：Charter 第 8 节「明确不做」的内容，规划中一律不出现，连预研都不排。

---

## 2. 阶段总览

第一阶段（MVP）拆为 **5 个里程碑（M0–M4）**，按依赖顺序串行推进。

```text
M0 项目基建与可观测骨架
  -> M1 仓库导入与原始构建判卷（无 AI）
  -> M2 覆盖率基线与目标选择
  -> M3 AI 生成与执行闭环（单轮，无修复）
  -> M4 失败归因、修复闭环与质量报告
```

关键判断：

- **M1、M2 完全不含 AI**。它们验证的是「判卷场」本身是否可信。判卷场不可信，AI 结果就无法被信任。
- **M3 才首次引入 LLM**，但只做单轮生成 + 执行，不修复。
- **M4 才闭合 Charter 第 11 节的完整流程**。

> 任何想把 AI 提前到 M1/M2 的做法，都违反「闭环优先」。判卷地基必须先于生成能力。

---

## 3. 里程碑详述

### M0 — 项目基建与可观测骨架

**目的**：让后续每个阶段都有地方「落结果、看轨迹」。

**范围**：

1. 仓库初始化、目录约定、运行约定。
2. 任务（Task）的最小生命周期定义：创建 / 运行中 / 成功 / 失败。
3. Agent 执行轨迹的步骤模型（对应 Charter 10.4 的 8 个步骤：Analyze / Context Collect / Scenario Plan / Generate / Execute / Fix / Validate / Report）。
4. 一个最小可见的 Web 骨架（能展示项目列表、任务列表、空轨迹页即可）。
5. 隔离执行目录的约定（每个任务一个独立工作目录，互不污染）。

**不做**：真实导入、真实构建、AI。

**产出**：可运行的空壳平台 + 能新建一个「假任务」并看到 8 步轨迹占位。

**对应 Charter**：10.1 / 10.4 页面骨架；为后续所有指标提供承载。

---

### M1 — 仓库导入与原始构建判卷（无 AI）

**目的**：证明「判卷场」能对**原始项目**做出可信判定。

**范围**：

1. 通过 Git 地址 + 分支导入一个开源 Java Maven 项目（单模块优先）。
2. 记录当前 commit、导入状态。
3. 在隔离目录执行原始 `mvn test`。
4. 解析 Surefire 测试结果（通过 / 失败 / 用例数）。
5. 在项目详情页展示：Maven 信息、src 路径、pom 信息、原始测试执行状态。

**不做**：覆盖率、目标选择、AI、修复。

**产出**：导入 ≥1 个真实开源 Maven 项目，并跑出原始 `mvn test` 结果与 Surefire 解析。

**对应 Charter MVP 标准**：第 1、2、3 条。
**对应 Charter 页面**：10.1、10.2。

**退出判据（验收）**：
- 能导入至少 1 个开源 Maven 项目。
- 能执行原始 `mvn test` 且结果可解析展示。
- 同一项目重复导入/执行结果稳定（隔离目录无污染）。

---

### M2 — 覆盖率基线与目标选择

**目的**：建立「改动前」的覆盖率基线，并让用户能指定判卷对象。

**范围**：

1. 集成 JaCoCo，采集原始覆盖率报告并解析。
2. 在详情页展示原始覆盖率（基线）。
3. 任务创建页：选择目标项目、目标类、目标方法、最大修复轮次（默认 3）、是否启用覆盖率门禁。
4. 目标上下文收集（Context Collect）：收集目标类/方法的源码、构造函数、依赖类型、返回值类型等生成所需信息。

**不做**：AI 生成、执行生成测试、修复。

**产出**：能解析 JaCoCo 基线覆盖率；能选定 1 个目标类或方法并收集其上下文。

**对应 Charter MVP 标准**：第 4、5、6 条。
**对应 Charter 页面**：10.2（覆盖率）、10.3（任务创建）。

**退出判据（验收）**：
- 能生成并解析 JaCoCo 覆盖率报告。
- 能选择 1 个目标类或方法。
- 能收集到足以喂给 LLM 的目标上下文（源码 + 依赖签名）。

---

### M3 — AI 生成与执行闭环（单轮，无修复）

**目的**：首次引入 LLM，打通「生成 → 写文件 → 执行」最窄链路。

**范围**：

1. Scenario Plan：基于上下文规划测试场景（happy path + 关键边界，最小集合）。
2. 调用 LLM 生成 JUnit5 + Mockito 测试文件。
3. 写入**独立测试文件**（不触碰生产代码）。
4. 执行生成后的 `mvn test`。
5. 记录执行结果与失败日志。
6. Patch 预览页展示新增测试文件，并标注「是否修改生产代码」。

**不做**：失败归因分类、自动修复、最终质量报告（留 M4）。

**产出**：对一个目标类，AI 生成测试文件 → 写入 → 执行 → 拿到结果（无论成败）。

**对应 Charter MVP 标准**：第 7、8、9、10 条。
**对应 Charter 页面**：10.4（Generate/Execute 步骤）、10.5（Patch 预览）。

**退出判据（验收）**：
- 能调用 LLM 生成 JUnit5 测试并写入独立文件。
- 能执行生成后的测试并拿到 Surefire 结果。
- 能记录失败日志。
- 全程未修改任何生产代码（可校验）。

---

### M4 — 失败归因、修复闭环与质量报告

**目的**：闭合 Charter 第 11 节完整流程，输出可审查的质量结论。

**范围**：

1. Coverage Compare：对比生成前后覆盖率（不下降 + 目标覆盖率提升）。
2. Failure Classify：对失败做归因（编译失败 / import 幻觉 / Mock 错误 / 断言失败 / 运行异常等基础分类）。
3. Fix Attempt：基于失败信息做有限轮次修复，**最多 3 轮**，每轮留 diff 轨迹。
4. Validate：修复后重新执行并校验。
5. Report：输出质量报告——编译是否通过、测试是否通过、覆盖率变化、断言质量、失败类型、修复轮次、**是否建议采纳**。
6. badcase 归档 + 基础规则知识库（把失败经验沉淀成规则）。
7. Patch 预览页补全：每轮修复 diff、最终 patch。

**不做**：Charter 第 8 节全部禁止项（自动入仓、PR、mutation testing、复杂 RAG 等）。

**产出**：对一个目标类完整跑完 Charter 第 11 节流程，产出最终 patch + 质量报告 + 采纳建议 + badcase 归档。

**对应 Charter MVP 标准**：第 11、12、13、14 条。
**对应 Charter 页面**：10.4（Fix/Validate/Report）、10.5（修复 diff/最终 patch）、10.6（质量报告）。

**退出判据（验收）**：
- 能对失败做归因。
- 能进行至少 1 轮、至多 3 轮修复并保留轨迹。
- 能输出最终 patch。
- 能输出质量报告并给出是否建议采纳。
- 能归档 badcase。

## 5. 北极星指标的过程校验点

Charter 第 6 节的 10 项指标，按里程碑设置校验点：

| 指标 | 首次可校验里程碑 |

| 原始项目是否可构建 | M1 |

| 失败是否有归因 | M4 |
| 修复是否有轨迹 | M4 |
| patch 是否可人工 review | M4 |

## 6. 关键依赖与顺序约束

1. **M1 必须先于一切 AI 工作**。判卷场不可信 → AI 结果无意义。
2. **M2 的覆盖率基线必须先于 M4 的覆盖率对比**。没有基线就无法判断「不下降 / 提升」。
3. **M3 的「独立测试文件 + 不碰生产代码」是硬约束**，一旦破坏，后续所有 patch 与采纳建议都不可信。
4. **M4 的修复轮次硬上限为 3**，超出即判失败并归因，不无限重试。
5. **隔离执行目录（M0 约定）**是所有「可重复、可信」结论的物理前提。

## 7. 风险登记册

| 风险 | 影响 | 应对（规划层面） |
|---|---|---|
| 开源项目环境复杂（依赖拉取慢/失败、JDK 版本不符） | M1 无法构建 | 选样阶段先人工筛 1–2 个「干净可构建」的单模块样板项目作为黄金样例 |
| 原始项目自身测试就失败 | 基线不可信 | 优先选原始 `mvn test` 全绿的项目；失败项目仅作扩展样例 |
| AI 生成编译不过 / import 幻觉（Charter 5.2） | M3 闭环卡死 | M2 的上下文收集要足够（依赖签名、构造函数）；失败计入 M4 归因，不在 M3 救火 |
| AI 改动生产代码 | 破坏判卷可信度 | M3 起把「未改生产代码」做成自动校验，违反即任务失败 |
| 覆盖率「虚高」（断言无效但覆盖率涨） | 采纳建议失真 | M4 把「断言质量」纳入报告，覆盖率提升不等于通过 |
| 修复进入无限循环 | 资源浪费 | 3 轮硬上限 + 每轮 diff 留痕，超限即归因归档 |
| 范围蔓延（被要求加 Gradle/多语言/自动入仓等） | 偏离 MVP | 以 Charter 第 8 节为红线，任何此类需求先改 Charter |

## 8. 选样计划（黄金样例）

MVP 期间至少准备样例项目，分两档：

- **A 档（必须）**：1 个单模块、依赖干净、原始 `mvn test` 全绿、有明确可被补测的目标类的开源 Maven 项目。作为 M1–M4 的主验收对象。
- **B 档（可选增强）**：1 个简单多模块项目，仅用于验证扩展性，不进 MVP 必需验收。

> 具体选哪个仓库，在 M1 启动时确定并记录到《选样记录》。

## 10. 本规划明确不排期的事项

以下来自 Charter 第 8 节，规划中**不立任务、不预研、不占工时**：

多语言、Gradle、IDE 插件、自动提交/PR/合并入仓、UI/接口自动化、手工用例、复杂 RAG、知识图谱、企业权限、多租户、微服务拆分、分布式调度、mutation testing、大规模并发生成、全仓库批量补测、生产代码自动修改。

如需其中任一项，先修改 Charter，再修改本规划。
