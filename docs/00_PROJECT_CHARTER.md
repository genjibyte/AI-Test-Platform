# TestAgent Lab：AI 单测生成判卷场

## 0. 文档目的

本文档是项目的最高约束文件。

所有 Coding Agent、后续架构设计、开发任务、技术选型、评测方案，都必须服从本文档。

如果后续需求与本文档冲突，必须先修改本文档，再修改代码。

本项目第一阶段不追求大而全，不追求企业级规模，不追求自动入仓。

第一阶段只追求一件事：

构建一个能验证 AI 生成单测是否可用的工程闭环。

---

## 1. 项目定位

本项目不是传统单测平台。

传统单测平台解决的是：

- 已有测试怎么执行；
- 覆盖率怎么看；
- CI 门禁怎么卡；
- 测试失败怎么展示。

本项目要解决的是：

当代码缺少单测或单测不足时，AI 能否生成可编译、可执行、可度量、可修复、可审查的测试补丁。

因此，本项目第一阶段准确定位为：

AI 单测生成判卷场。

也可以描述为：

面向开源 Java Maven 项目的 AI 单元测试生成、执行判卷、失败修复与质量治理系统。

---

## 2. 一句话介绍

输入一个 Git 仓库、分支、目标 Java 类或方法，系统自动收集上下文，生成 JUnit5 + Mockito 单测，在隔离环境中执行 Maven Test 和 JaCoCo 覆盖率统计，失败后进行有限轮次修复，最终输出测试 patch、覆盖率变化、失败归因、修复轨迹和是否建议采纳的质量报告。

---

## 3. 第一阶段核心判断

第一阶段必须坚持以下判断：

1. 单测方向服务的第一用户是开发人员，不是业务测试人员。
2. 项目由测开 / 质量效能视角建设，但不是替测试人员写黑盒用例。
3. 第一阶段模拟的是大厂里的白盒研发效能场景。
4. 开源仓库可以模拟企业代码仓库，但不能模拟全部企业测试资产。
5. 本项目不复制快手、Uber 的终局平台，只复刻它们的最小可靠闭环。
6. 第一阶段最大价值不是“生成测试”，而是“判定 AI 生成测试是否工程可用”。

---

## 4. 目标用户

第一阶段目标用户包括：

### 4.1 开发工程师

目标：

- 减少手写单测成本；
- 获得可 review 的测试 patch；
- 在重构、改动、补测时获得质量辅助。

### 4.2 测试开发 / 质量效能工程师

目标：

- 建设 AI 单测生成能力；
- 管理覆盖率、质量门禁、失败归因；
- 沉淀 badcase 和修复规则。

### 4.3 研发负责人

目标：

- 观察 AI 生成单测的可用率；
- 观察增量代码覆盖率变化；
- 观察失败类型分布；
- 评估 AI 测试生成是否值得接入研发流程。

### 4.4 CI/CD 平台

第一阶段不直接接入 CI，但架构需要预留：

- PR 阶段自动补测；
- 生成建议 patch；
- 输出质量报告；
- 不自动入仓。

---

## 5. 要解决的真实痛点

### 5.1 传统单测痛点

1. 开发写单测成本高，容易被业务交付挤压。
2. 存量代码覆盖不足，重构和修改缺少安全网。
3. 传统单测生成插件偏模板化，断言、Mock、边界场景仍依赖人工。
4. 覆盖率平台只能度量已有测试，不能解决测试资产不足。

### 5.2 AI 生成单测的新痛点

1. AI 生成代码看起来正确，但经常编译不过。
2. AI 容易产生 import 幻觉。
3. AI 不理解构造函数、泛型、依赖类型、返回值类型。
4. AI 容易生成错误 Mock。
5. AI 倾向生成 happy path，边界和异常场景不足。
6. AI 生成后缺少真实执行验证。
7. AI 失败后没有系统化修复闭环。
8. 覆盖率提升不等于测试有效，需要质量门禁。
9. 失败经验没有沉淀，下一次仍然重复踩坑。
10. 普通 Coding Agent 更像代码生成器，不像质量系统。

---

## 6. 第一阶段北极星指标

第一阶段只关注闭环可信度，不关注规模化。

北极星指标：

AI 生成的测试是否经过真实工程判卷，并形成可审查结果。

具体指标：

1. 原始项目是否可构建。
2. 生成测试是否可编译。
3. 生成测试是否可执行。
4. 覆盖率是否不下降。
5. 目标类或目标方法覆盖率是否提升。
6. 断言是否有效。
7. 是否没有修改生产代码。
8. 失败是否有归因。
9. 修复是否有轨迹。
10. patch 是否可人工 review。

---

## 7. 第一阶段范围

第一阶段只支持以下能力：

1. Java。
2. Maven。
3. JUnit5。
4. Mockito。
5. Git 仓库导入。
6. 单模块 Maven 项目优先。
7. 简单多模块 Maven 项目可作为增强项，不作为 MVP 必需项。
8. 指定类 / 方法级生成。
9. 生成独立测试文件。
10. 本地隔离目录执行 `mvn test`。
11. JaCoCo 覆盖率采集。
12. Surefire 测试结果解析。
13. 最多 3 轮失败修复。
14. 输出 patch。
15. 输出质量报告。
16. 记录 badcase。
17. 基础规则知识库。

---

## 8. 第一阶段明确不做

以下内容第一阶段禁止实现：

1. 不做多语言支持。
2. 不做 Gradle 支持。
3. 不做 IDE 插件。
4. 不做自动提交代码。
5. 不做自动创建 PR/MR。
6. 不做自动合并入仓。
7. 不做 UI 自动化。
8. 不做接口自动化。
9. 不做手工用例生成。
10. 不做复杂 RAG。
11. 不做知识图谱。
12. 不做企业权限系统。
13. 不做多租户。
14. 不做微服务拆分。
15. 不做分布式任务调度。
16. 不做 mutation testing。
17. 不做大规模并发生成。
18. 不做全仓库批量补测。
19. 不做生产代码自动修改。

这些能力可以进入 Roadmap，但不得进入 MVP。

## 9. MVP 成功标准

MVP 成功不要求高采纳率。

MVP 成功只要求闭环成立。

MVP 必须满足：

1. 可以导入至少 1 个开源 Maven 项目。
2. 可以执行原始 `mvn test`。
3. 可以生成并解析 Surefire 测试结果。
4. 可以生成并解析 JaCoCo 覆盖率报告。
5. 可以选择 1 个目标类或方法。
6. 可以收集目标上下文。
7. 可以调用 LLM 生成 JUnit5 测试文件。
8. 可以写入独立测试文件。
9. 可以执行生成后的测试。
10. 可以记录失败日志。
11. 可以进行至少 1 轮修复。
12. 可以输出最终 patch。
13. 可以输出质量报告。
14. 可以判断是否建议采纳。

## 10. 第一阶段产品形态

第一阶段优先做 Web 平台。

不做 IDE 插件。

最小页面包括：

### 10.1 项目列表页

展示：

- 项目名称；
- Git 地址；
- 分支；
- 当前 commit；
- 导入状态；
- 最近一次测试状态。

### 10.2 项目详情页

展示：

- Maven 项目信息；
- src/main/java 路径；
- src/test/java 路径；
- pom.xml 信息；
- 原始测试执行状态；
- 原始覆盖率。

### 10.3 任务创建页

输入：

- 目标项目；
- 目标类；
- 目标方法；
- 最大修复轮次，默认 3；
- 是否启用覆盖率门禁。

### 10.4 Agent 执行轨迹页

展示每一步：

- Analyze；
- Context Collect；
- Scenario Plan；
- Generate；
- Execute；
- Fix；
- Validate；
- Report。

每一步展示：

- 状态；
- 输入摘要；
- 输出摘要；
- 耗时；
- 错误原因；
- 日志入口。

### 10.5 Patch 预览页

展示：

- 新增测试文件；
- 每轮修复 diff；
- 最终 patch；
- 是否修改生产代码。

### 10.6 质量报告页

展示：

- 编译是否通过；
- 测试是否通过；
- 覆盖率变化；
- 断言质量；
- 失败类型；
- 修复轮次；
- 是否建议采纳；
- badcase 归档结果。

---

## 11. 核心流程

第一阶段流程固定为：

```text
Repo Import
  -> Build Check
  -> Coverage Baseline
  -> Target Select
  -> Context Collect
  -> Scenario Plan
  -> Test Generate
  -> Test Execute
  -> Coverage Compare
  -> Failure Classify
  -> Fix Attempt
  -> Validate
  -> Report
```
